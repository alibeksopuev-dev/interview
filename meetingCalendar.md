# 📅 Meeting Calendar — Глубокий разбор задачи

> **Сложность:** Easy &nbsp;|&nbsp; **Язык:** TypeScript / JavaScript &nbsp;|&nbsp; **Время:** ~35 мин

---

## 📋 Условие задачи

Дан массив временных интервалов встреч `intervals`, где каждый элемент представляет встречу в виде `[start, end]`.

**Необходимо определить:** может ли человек посетить **все** встречи, не имея пересечений по времени?

### Входные данные

| Параметр | Тип | Описание |
|---|---|---|
| `intervals` | `number[][]` | 2D массив размером `n × 2` |
| `intervals[i][0]` | `number` | Время **начала** i-й встречи |
| `intervals[i][1]` | `number` | Время **конца** i-й встречи |

### Ограничения

```
0 <= intervals.length <= 1000
intervals[i].length == 2
0 <= start < end <= 1_000_000
```

---

## 💡 Примеры

### Пример 1 — одна встреча
```
Input:  intervals = [[83, 99]]
Output: true
```
> Одна встреча — нет возможности для пересечения.

---

### Пример 2 — встречи «встык»
```
Input:  intervals = [[1,5], [5,10], [10,15]]
Output: true
```
> Встречи идут одна за другой без перекрытия. Граница не считается пересечением.

---

### Пример 3 — есть пересечение
```
Input:  intervals = [[8,10], [1,3], [2,6], [15,18]]
Output: false
```
> Встречи `[1, 3]` и `[2, 6]` пересекаются — в момент времени `2` человек должен быть на двух встречах одновременно.

---

## 🔴 Подход 1: Грубая сила (Brute Force)

### Идея

Перебрать **все возможные пары** интервалов и проверить, не пересекаются ли они.  
Если хотя бы одна пара пересекается — вернуть `false`.

### Алгоритм

1. Определить вспомогательную функцию `overlap(interval1, interval2)`
2. Использовать **двойной цикл** для перебора всех пар `(i, j)`
3. Для каждой пары вызвать `overlap` → если `true`, вернуть `false`
4. Если ни одной пересекающейся пары не найдено → вернуть `true`

### Код

```typescript
export function isMeetingCalendarValidBruteForce(intervals: number[][]): boolean {

  // Шаг 1: Вспомогательная функция проверки пересечения
  // Два интервала [a,b] и [c,d] пересекаются, если:
  //   • a >= c И a < d  →  начало первого внутри второго
  //   • ИЛИ c >= a И c < b  →  начало второго внутри первого
  // Граница НЕ пересечение: [1,5] и [5,10] → 5 < 5 = false → ОК
  const overlap = (interval1: number[], interval2: number[]): boolean => {
    return (
      (interval1[0] >= interval2[0] && interval1[0] < interval2[1]) ||
      (interval2[0] >= interval1[0] && interval2[0] < interval1[1])
    );
  };

  // Шаг 2: Двойной цикл по всем парам
  // j = i + 1: не сравниваем интервал сам с собой и не дублируем пары
  for (let i = 0; i < intervals.length; i++) {
    for (let j = i + 1; j < intervals.length; j++) {

      // Шаг 3: Проверка текущей пары
      if (overlap(intervals[i], intervals[j])) {
        return false; // Пересечение найдено → calendar НЕ валиден
      }
    }
  }

  // Шаг 4: Пересечений нет
  return true;
}
```

### Big-O анализ

| Сложность | Обозначение | Пояснение |
|---|---|---|
| **Время** | **O(n²)** | Перебираем все `n*(n-1)/2` пар |
| **Память** | **O(1)** | Дополнительных структур не используем |

## 🟢 Подход 2: С сортировкой (Optimal)

### Идея

После **сортировки** по времени начала, перекрывающиеся интервалы гарантированно окажутся рядом. Достаточно проверять только **соседние пары** — линейный проход вместо квадратичного.

### Алгоритм

1. Отсортировать `intervals` по времени начала (возрастание)
2. Итерироваться по отсортированным интервалам
3. Для каждого `i` проверить: `intervals[i][1] > intervals[i+1][0]`
4. Если условие истинно → вернуть `false`
5. Цикл завершился без пересечений → вернуть `true`

### Код

```typescript
export default function isMeetingCalendarValid(intervals: number[][]): boolean {

  // Шаг 1: Сортировка по времени начала
  // a[0] - b[0] → возрастающий порядок
  //
  // Пример:
  //   До:    [[8,10],[1,3],[2,6],[15,18]]
  //   После: [[1,3],[2,6],[8,10],[15,18]]
  intervals.sort((a, b) => a[0] - b[0]);

  // Шаг 2: Линейный проход
  // Цикл до length-1: сравниваем [i] со [i+1], последний не имеет "следующего"
  for (let i = 0; i < intervals.length - 1; i++) {

    // Шаг 3: Проверка соседней пары
    // intervals[i][1]   — конец текущей встречи
    // intervals[i+1][0] — начало следующей встречи
    //
    // Граничный случай (встречи "встык"):
    //   [1,5] и [5,10] → 5 > 5 = false → НЕ пересечение ✓
    if (intervals[i][1] > intervals[i + 1][0]) {
      return false; // Пересечение найдено
    }
  }

  // Шаг 4: Конфликтов нет
  return true;
}
```

### Big-O анализ

| Сложность | Обозначение | Пояснение |
|---|---|---|
| **Время** | **O(n log n)** | Сортировка доминирует; линейный проход O(n) |
| **Память** | **O(1)** | Сортировка in-place, доп. память не нужна |

## ⚖️ Сравнение алгоритмов

| Подход | Время | Память | Когда использовать |
|---|---|---|---|
| **Brute Force** | O(n²) | O(1) | Малые данные, простота реализации |
| **Sorting (оптимальный)** | O(n log n) | O(1) | Production, большие данные |

### Почему сортировка лучше?

При `n = 1000`:

```
Brute Force:  n*(n-1)/2 = 499 500 операций
Sorting:      n*log₂(n) ≈ 9 966 операций
```

> Разница в **~50 раз** при `n = 1000` и растёт по мере увеличения `n`.

---

## 🔑 Ключевые выводы

> [!NOTE]
> **Граница не считается пересечением:** встречи `[1, 5]` и `[5, 10]` считаются допустимыми — проверка строгая (`>`), а не нестрогая (`>=`).

> [!TIP]
> **Сортировка как инструмент:** если задача требует сравнения соседних элементов или гарантирует, что конфликты расположены рядом — сортировка почти всегда оптимизирует решение с O(n²) до O(n log n).

> [!IMPORTANT]
> **Сортировка мутирует массив:** `intervals.sort(...)` изменяет исходный массив in-place. Если исходные данные нужно сохранить — используйте `[...intervals].sort(...)`.
