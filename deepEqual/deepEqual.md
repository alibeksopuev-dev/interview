# Разбор алгоритма `deepEqual` (Big O + пошаговый стек)

> Как сравнивать вложенные объекты рекурсивно и почему мы сначала обрабатываем примитивы.

---

## 📌 Введение: в чём идея?

Представь, что у тебя есть два объекта: `{ id: 1 }` и `{ id: 1 }`. Если сравнить их через `===`, JavaScript вернёт
`false`. Почему? Потому что это два **разных объекта в памяти** (разные ссылки).

Для того чтобы узнать, равны ли они **по значению** (содержимому), нам нужно обойти их рекурсивно.

**Альтернативный подход (Handling primitives first):** Сначала мы проверяем значения с помощью `Object.is`. Если они
полностью идентичны (включая ссылки или примитивы), то сразу возвращаем `true` и избегаем лишней работы. Если нет, мы
убеждаемся, что оба значения — это **перебираемые контейнеры** (оба объекты или оба массивы). Затем проверяем совпадение
количества их ключей и рекурсивно сравниваем каждое свойство.

---

## ⏱️ 1. Временная сложность (Time Complexity): `O(N)`

Здесь `N` — это суммарное количество всех свойств (и под-свойств) в сравниваемых объектах.

### Почему `O(N)`?

Чтобы убедиться, что два объекта полностью идентичны, мы обязаны посетить **каждое свойство** обоих объектов хотя бы
один раз.

- В худшем случае (когда объекты глубоко равны) мы обойдем все `N` элементов дерева.
- Если объекты различаются, алгоритм прервётся раньше (early exit благодаря `for...in` с `return false`), но в оценке
  Big O мы всегда смотрим на худший сценарий.
- Вызов `Object.keys()` требует `O(K)` времени на каждый узел (где `K` — количество ключей узла). В сумме по всем узлам
  это даёт `O(N)`.

Поэтому временная сложность строго линейная: **`O(N)`**.

---

## 💾 2. Пространственная сложность (Space Complexity): `O(D)`

Здесь `D` — максимальная глубина вложенности объектов (Depth).

1. **Стек вызовов:** Каждый раз, когда мы встречаем вложенный объект или массив, функция `deepEqual` вызывает саму себя
   (рекурсия). Это добавляет кадр в стек вызовов. Если глубина матрешки из объектов равна 10, стек будет глубиной 10. То
   есть `O(D)`.
2. **Память на массивы ключей:** На каждом уровне вызов `Object.keys()` создаёт новый массив ключей памяти. Обычно это
   требует минимальной памяти на каждом уровне.

Поэтому корректнее всего оценивать сложность по глубине рекурсии: **`O(D)`**. Если структура выглядит как "колбаса"
(один объект внутри другого, и так до конца), то `D = N`, и память станет `O(N)`.

---

## 🔬 3. Пошаговый разбор: как работает стек вызовов

### Входные данные:

```typescript
valueA = [{ id: '1' }]
valueB = [{ id: '2' }]
```

Это массивы, внутри которых лежат объекты.

---

### 🔁 Шаг 1: Сравнение массивов (уровень 0)

Стек: `[ deepEqual(arrA, arrB) ]`

- Вызов `Object.is(arrA, arrB)` вернёт `false` (разные ссылки в памяти).
- Переменные `bothObjects = false`, `bothArrays = true`.
- Значения проходят через условие `if (!bothObjects && !bothArrays)` (оно равно false).
- Проверяем `Object.keys`: у обоих длина ключей `1` (индекс `'0'`).
- Запускаем цикл `for...in`. Переменная `key` = `'0'`.
- Рекурсивно вызываем `deepEqual` для `arrA['0']` и `arrB['0']`.

---

### 🔁 Шаг 2: Сравнение вложенных объектов (уровень 1)

Стек: `[ deepEqual(arrA, arrB)  ->  deepEqual(objA, objB) ]`

- Переданы значения: `{ id: '1' }` и `{ id: '2' }`.
- Вызов `Object.is` вернёт `false`.
- Переменные `bothObjects = true`, `bothArrays = false`.
- Проходим проверки типов.
- Проверяем длину: у обоих длина `1` (ключ `'id'`).
- Запускаем цикл `for...in`. Переменная `key` = `'id'`.
- Вызываем `deepEqual` для `objA['id']` и `objB['id']`.

---

### 🔁 Шаг 3: Сравнение примитивов (уровень 2)

Стек: `[ deepEqual(arrA, arrB)  ->  deepEqual(objA, objB)  ->  deepEqual('1', '2') ]`

- Переданы примитивы: `'1'` и `'2'`.
- Вызов `Object.is('1', '2')` возвращает `false`.
- Переменные `bothObjects = false`, `bothArrays = false`.
- Срабатывает условие `if (!bothObjects && !bothArrays)`.
- Возвращаем `false`.

---

### ↩️ Шаг 4: Возврат по стеку (схлопывание рекурсии)

1. Уровень 2 (`deepEqual('1', '2')`) завершается и возвращает `false`.
2. Уровень 1 (цикл для объектов) получает `false`, прерывается и делает `return false`.
3. Уровень 0 (цикл для массивов) получает `false`, прерывается и делает `return false`.

**Итог:** `deepEqual([{ id: '1' }], [{ id: '2' }]) === false`.

---

## 💡 4. Разбор неочевидных строк кода (FAQ)

### Почему подход "Сначала примитивы" (Approach 2) лучше, чем "Сначала объекты" (Approach 1)?

Существует альтернативный вариант решения (Approach 1), который сначала проверяет, являются ли значения
объектами/массивами, сразу делает рекурсию, а проверку примитивов и одинаковых ссылок оставляет на самый конец:

```typescript
// ❌ Менее эффективный вариант реализации (Approach 1)
export default function deepEqual(valueA: unknown, valueB: unknown): boolean {
  const type1 = getType(valueA)
  const type2 = getType(valueB)

  if (type1 === type2 && shouldDeepCompare(type1) && shouldDeepCompare(type2)) {
    const entriesA = Object.entries(valueA as Array<unknown> | Object)
    const entriesB = Object.entries(valueB as Array<unknown> | Object)

    if (entriesA.length !== entriesB.length) return false

    return entriesA.every(
      ([key, value]) => Object.hasOwn(valueB as Array<unknown> | Object, key) && deepEqual(value, (valueB as any)[key]),
    )
  }

  // Fallback (проверка примитивов и ссылок) только в самом конце!
  return Object.is(valueA, valueB)
}
```

Наш текущий подход ("Сначала примитивы" / Approach 2) **значительно эффективнее** по двум причинам:

1. **Фатальный недостаток производительности (O(N) вместо O(1) для одинаковых ссылок):** В Approach 1 проверка
   `Object.is()` стоит в самом конце как `fallback`. Из-за этого нет **"раннего выхода" (early exit)** для одинаковых
   объектов. Если ты вызовешь `deepEqual(hugeObject, hugeObject)`, Approach 1 пойдёт рекурсивно распаковывать **все**
   свойства огромного объекта до самых низов, чтобы сравнить каждый примитив. Наш алгоритм увидит одинаковую ссылку
   через `Object.is` на первой же строчке и мгновенно вернёт `true`.
2. **Лишняя нагрузка на сборщик мусора (Garbage Collection):** В Approach 1 используется `Object.entries()`, который на
   каждом шаге рекурсии создаёт новые массивы `[[ключ, значение], [ключ, значение]]`. На глубоких структурах это
   засоряет оперативную память и заставляет GC работать активнее. Наш цикл `for...in` таких "паразитных" структур не
   создаёт.

---

### Зачем нужно условие отсечения `if (!bothObjects && !bothArrays)`?

К моменту, когда код доходит до этой строчки, `Object.is(valueA, valueB)` уже вернул `false`. Это означает, что
переменные **не являются одним и тем же примитивом** и не ссылаются на один и тот же объект в памяти.

Но они всё ещё могут быть _глубоко равны_, только если **оба** являются объектами (`bothObjects === true`) или **оба**
являются массивами (`bothArrays === true`). Условие `(!bothObjects && !bothArrays)` буквально переводится как: _"Если
это НЕ два объекта И это НЕ два массива — значит, сравнивать тут больше нечего, они точно разные"_.

**Примеры, когда это отсекает лишнее:**

1. **Разные примитивы** (`'hello'` и `'world'`): Это не объекты и не массивы. Условие выполняется -> `false`.
2. **Примитив и объект** (`42` и `{ id: 42 }`): Оба объекты? Нет. Оба массивы? Нет. Условие выполняется -> `false`.
3. **Объект и массив** (`{ 0: 1 }` и `[1]`): В JS они похожи, но один объект, а другой массив. Условие выполняется ->
   `false`.
4. **Сравнение с `null`** (`null` и `{}`): `typeof null` это `'object'`, но наша строгая проверка отсеет `null`. Условие
   выполняется -> `false`.

---

### Зачем мы приводим типы к `Record<string, unknown>`?

В коде есть строки:

```typescript
const comparableA = valueA as Record<string, unknown>
const comparableB = valueB as Record<string, unknown>
```

Эти строки нужны **только для TypeScript** (на работу JS не влияют).

Изначально параметры имеют тип `unknown`. TypeScript запрещает обращаться к свойствам `unknown` (нельзя написать
`valueA[key]`). Хотя выше мы уже доказали кодом, что переменные точно являются объектами или массивами, TypeScript не
всегда запоминает этот вывод автоматически для сложных перекрёстных условий.

`as Record<string, unknown>` — это наша "клятва" компилятору: _"Я проверил, относись к этой переменной как к словарю
(объекту), у которого ключи — строки, а значения неизвестны"_. Это позволяет нам использовать обращение
`comparableA[key]` без красных подчёркиваний.

---

### Может ли `Record<string, unknown>` быть массивом?

**Да, абсолютно!** В JavaScript массив под капотом — это просто специфичный объект, у которого индексы являются
строковыми ключами. Массив `['apple', 'banana']` под капотом выглядит примерно так:

```javascript
{
  '0': 'apple',
  '1': 'banana',
  length: 2
}
```

Поэтому:

1. TypeScript позволяет привести массив к типу `Record<string, unknown>` (ведь это объект со строковыми ключами).
2. `Object.keys(['a', 'b'])` вернёт массив строк `['0', '1']`.
3. Цикл `for...in` переберёт эти строковые ключи (`'0'`, `'1'`).
4. Обращение `comparableA['0']` сработает для массива точно так же, как обычный доступ по индексу.

Приведение к `Record` — это удобный трюк, чтобы одним и тем же кодом одинаково обрабатывать обход и для массивов, и для
объектов.

---

### Как работает связка `entries.every` и `Object.hasOwn` в Approach 1?

В альтернативном варианте (Approach 1) для обхода свойств используется такая конструкция:

```typescript
return entriesA.every(([key, value]) => Object.hasOwn(valueB, key) && deepEqualApproach1(value, valueB[key]))
```

Разберём её пошагово:

1. **`.every()`**: Берёт каждый элемент массива `entriesA` по очереди. Если условие выполняется для _каждого_ элемента,
   возвращает `true`. Если хотя бы для одного даст `false` — моментально останавливается и возвращает `false`.
2. **`([key, value]) =>`**: "Распаковывает" (деструктурирует) каждую пару из массива `entries` (например, `['id', 42]`)
   в две удобные переменные.
3. **`Object.hasOwn(valueB, key)`**: Проверяет, _"а есть ли вообще такой ключ во втором объекте?"_. Если ключа нет,
   проверка сразу падает (возвращает `false`).
4. **`&& deepEqualApproach1(...)`**: Если ключ всё-таки есть, мы закидываем значения из обоих объектов обратно в
   "мясорубку" `deepEqualApproach1` (рекурсия), чтобы убедиться, что их содержимое тоже полностью совпадает.

Этот кусок кода буквально означает: _"Пройдись по всем свойствам первого объекта. Если для каждого свойства во втором
объекте есть такой же ключ И их значения идентичны — объекты равны. Иначе — они разные."_

## ⚠️ 5. Edge Cases (Граничные случаи)

При реализации такого простого обхода нужно помнить о его ограничениях:

1. **`null` и `Objects`:** `typeof null` возвращает `'object'`, поэтому мы используем `Object.prototype.toString.call` —
   он чётко отличает реальный объект (`[object Object]`) от `null` (`[object Null]`).
2. **Равенство `+0` и `-0`:** Мы используем `Object.is`, который считает их **неравными**. Обычный `===` посчитал бы их
   равными. Аналогично `NaN` и `NaN` (в `Object.is` они равны).
3. **Циклические объекты:** Если объект ссылается сам на себя (Circular reference), алгоритм уйдёт в бесконечную
   рекурсию и выбросит ошибку `Maximum call stack size exceeded`.
4. **Дескрипторы свойств:** Такие вещи, как `configurable`, `writable` или `get/set`, игнорируются при сравнении.
5. **Неперечисляемые свойства и Symbols:** `for...in` и `Object.keys()` обходят **только перечисляемые
   свойства-строки**. Свойства с `enumerable: false` или ключи, созданные через `Symbol`, будут полностью
   проигнорированы.
