# Разбор алгоритма `binarySearch` (Big O + пошаговый стек)
> Как правильно "разрезать" задачу пополам — и почему `O(log n)` это почти магия.

---

## 📌 Введение: в чём идея?

Представь, что тебе нужно найти слово в бумажном словаре из 1000 страниц.

**Наивный подход (линейный поиск):** листаешь с первой страницы — 1, 2, 3... Если слово на 987-й странице — ты листаешь 987 раз.

**Бинарный поиск:** открываешь словарь на 500-й странице. Слово там — *позже*? Тогда левые 500 страниц ты выбрасываешь мусорную корзину **навсегда**. Открываешь середину оставшихся. Снова выбрасываешь половину. И так далее.

Ключевое правило: **массив должен быть отсортирован**. Без сортировки трюк не работает — мы не знали бы, в какую сторону идти.

В коде это реализовано через три переменные-границы: `left`, `right`, и вычисляемый `mid`.

---

## ⏱️ 1. Временная сложность (Time Complexity): `O(log n)`

### Почему не `O(n)`?

В каждой итерации цикла `while` мы делаем одно сравнение — и после него **отбрасываем ровно половину оставшегося массива**. Навсегда.

```
Было 1000 элементов.
После итерации 1: осталось 500.
После итерации 2: осталось 250.
После итерации 3: осталось 125.
...
```

Вопрос: сколько раз нужно делить 1000 на 2, чтобы получить 1?

```
1000 → 500 → 250 → 125 → 62 → 31 → 15 → 7 → 3 → 1
```

Это ровно **10 шагов**. Математически: `log₂(1000) ≈ 10`.

Вот откуда `O(log n)`.

| Размер массива `n` | Макс. итераций бинарного поиска | Макс. итераций линейного поиска |
|--------------------|---------------------------------|---------------------------------|
| 10                 | 4                               | 10                              |
| 1 000              | 10                              | 1 000                           |
| 1 000 000          | 20                              | 1 000 000                       |
| 1 000 000 000      | 30                              | 1 000 000 000                   |

Миллиард элементов — **30 сравнений**. Это и есть логарифм.

### Что происходит внутри цикла?

```typescript
const mid = left + Math.floor((right - left) / 2); // O(1)
const midValue = arr[mid];                          // O(1) — обращение по индексу
if (midValue === target) { ... }                    // O(1)
```

Каждый шаг внутри цикла — мгновенные операции `O(1)`. Умножаем:

```
log(n) итераций × O(1) на каждую = O(log n)
```

> 🏆 **Почему `mid = left + Math.floor((right - left) / 2)`, а не просто `(left + right) / 2`?**  
> В языках с фиксированным размером целых чисел (C, Java) `left + right` может вызвать **переполнение** (integer overflow), если оба числа очень большие. В JavaScript это не критично из-за `Number`, но запись `left + (right - left) / 2` — правильная инженерная привычка.

## 💾 2. Пространственная сложность (Space Complexity): `O(1)`

Смотрим на всё, что мы создаём:

```typescript
let left = 0;           // одна переменная-число
let right = arr.length - 1; // одна переменная-число
// внутри цикла:
const mid = ...;        // одна переменная-число (пересоздаётся каждый раз, не накапливается)
const midValue = ...;   // одна переменная-число
```

Итого: **4 переменные**, и это число не меняется вне зависимости от размера входного массива.

Мы не создаём новые массивы. Мы не копируем данные. Мы просто двигаем три числа по уже существующему массиву.

```
Массив из 7 элементов:   4 переменные в памяти.
Массив из 7 000 000:     4 переменные в памяти.
```

> 🏆 **`O(1)` — константная память.** Алгоритм "смотрит" на массив, но ничего из него не копирует.

## 🔬 3. Пошаговый разбор: как меняются переменные

### Входные данные:
```typescript
arr    = [1, 3, 5, 7, 9, 11, 13]
//  idx:  0  1  2  3  4   5   6
target = 7
```

Длина массива `n = 7`. Стартуем: `left = 0`, `right = 6`.

---

### 🔁 Итерация 1

```
Состояние:  left=0, right=6
Диапазон:   [1, 3, 5, 7, 9, 11, 13]
             ^                    ^
```

```typescript
mid      = 0 + Math.floor((6 - 0) / 2) = 0 + 3 = 3
midValue = arr[3] = 7
```

```
arr:  [ 1,  3,  5, [7],  9, 11, 13 ]
                    ^
                   mid
```

`midValue (7) === target (7)` ✅ — **НАШЛИ!**

Возвращаем `mid = 3`. Готово.

---

### Второй пример: `target = 11` (элемент в правой половине)

Те же данные: `arr = [1, 3, 5, 7, 9, 11, 13]`, `left = 0`, `right = 6`.

---

#### 🔁 Итерация 1

```
Состояние:  left=0, right=6
Диапазон:  [1, 3, 5, 7, 9, 11, 13]
            ^                    ^
           left                right
```

```typescript
mid      = 0 + Math.floor((6 - 0) / 2) = 3
midValue = arr[3] = 7
```

`midValue (7) < target (11)` → цель **правее**.

```typescript
left = mid + 1 = 4   // левую половину [0..3] выбрасываем навсегда
```

---

#### 🔁 Итерация 2

```
Состояние:  left=4, right=6
Диапазон:  [_, _, _, _, 9, 11, 13]
                        ^       ^
                       left   right
```

```typescript
mid      = 4 + Math.floor((6 - 4) / 2) = 4 + 1 = 5
midValue = arr[5] = 11
```

`midValue (11) === target (11)` ✅ — **НАШЛИ!**

Возвращаем `mid = 5`. Всего **2 итерации** из возможных 7 при линейном поиске.

---

### Третий пример: `target = 6` (элемента нет в массиве)

`arr = [1, 3, 5, 7, 9, 11, 13]`, `left = 0`, `right = 6`.

---

#### 🔁 Итерация 1

```
Состояние:  left=0, right=6
mid = 3, midValue = arr[3] = 7
```

`midValue (7) > target (6)` → цель **левее**.

```typescript
right = mid - 1 = 2
```

---

#### 🔁 Итерация 2

```
Состояние:  left=0, right=2
Диапазон:  [1, 3, 5, _, _, _, _]
            ^       ^
           left   right
```

```typescript
mid      = 0 + Math.floor((2 - 0) / 2) = 1
midValue = arr[1] = 3
```

`midValue (3) < target (6)` → цель **правее**.

```typescript
left = mid + 1 = 2
```

---

#### 🔁 Итерация 3

```
Состояние:  left=2, right=2
Диапазон:  [_, _, 5, _, _, _, _]
                  ^
               left = right = mid
```

```typescript
mid      = 2 + Math.floor((2 - 2) / 2) = 2
midValue = arr[2] = 5
```

`midValue (5) < target (6)` → цель **правее**.

```typescript
left = mid + 1 = 3
```

---

#### ❌ Выход из цикла

```
Состояние:  left=3, right=2
Условие while (left <= right): 3 <= 2 — FALSE
```

Диапазон схлопнулся. `left` перепрыгнул через `right`. Элемент не найден.

Возвращаем `-1`.

## 🥊 Сравнение: Binary Search vs Linear Search

```typescript
// Линейный поиск — O(n)
const linearSearch = (arr: number[], target: number): number => {
  for (let i = 0; i < arr.length; i++) {
    if (arr[i] === target) return i;
  }
  return -1;
};
```

| Критерий                     | Линейный поиск   | Бинарный поиск        |
|------------------------------|------------------|-----------------------|
| Временная сложность          | `O(n)`           | `O(log n)` 🏆         |
| Пространственная сложность   | `O(1)`           | `O(1)` 🤝             |
| Требует сортировки?          | ❌ Нет           | ✅ Да — обязательно   |
| Подходит для несортированных | ✅ Да            | ❌ Нет                |
| 1 млрд элементов: итераций   | до 1 000 000 000 | до 30 🤯              |

> **Вывод:** бинарный поиск — это инструмент с одним жёстким контрактом: *"дай мне отсортированный массив, и я найду что угодно за 30 шагов, даже в миллиарде элементов"*.
