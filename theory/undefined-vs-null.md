# Разница между `undefined` и `null`

Оба этих слова в JavaScript означают "пустоту" или "отсутствие значения", но между ними есть принципиальная разница в
том, **кто именно** создал эту пустоту.

Давай представим, что переменная — это подарочная коробка.

## 1. `undefined` (Неопределено)

Это пустота, которую создаёт **сам JavaScript**. Означает: _"Коробка существует, но в неё ещё никто ничего не положил"_.

Если ты создашь переменную, но забудешь дать ей значение, JavaScript автоматически положит туда `undefined`. То же самое
происходит, если функция ничего не возвращает, или если ты пытаешься прочитать свойство объекта, которого там отродясь
не было.

```javascript
let box
console.log(box) // undefined (ты просто создал коробку, она пока пуста)

const user = { name: 'John' }
console.log(user.age) // undefined (такого ключа в объекте нет)

function doNothing() {}
console.log(doNothing()) // undefined (функция ничего не вернула)
```

## 2. `null` (Ничто / Ноль)

Это пустота, которую создаёт **сам программист** (ты). Означает: _"Я, как разработчик, специально говорю, что здесь
ПУСТО. Коробка пуста по моему осознанному решению"_.

Мы используем `null`, когда хотим стереть старое значение или показать, что данных осознанно нет (например, пользователь
ещё не загрузил аватарку, и мы помечаем это как `null`).

```javascript
let currentManager = 'Alice'
// Алису уволили, менеджера пока нет
currentManager = null
```

## 📋 Краткий итог (Шпаргалка)

- **`undefined`** = "Ещё не назначено" (Оставил пустым JavaScript или по умолчанию). _Как пустая графа в анкете, которую
  ты просто пропустил._
- **`null`** = "Специально пусто" (Очистил программист). _Как графа в анкете, где ты ручкой написал "Не применимо /
  Отсутствует"._

---

> 🐞 **Забавный баг JS (typeof null)** Если спросить у JS `typeof undefined`, он логично ответит `'undefined'`. А если
> спросить `typeof null`, он ответит `'object'`, хотя `null` объектом не является. Это баг из самых первых версий
> JavaScript (1995 год), который так и не исправили, чтобы не сломать старые сайты в интернете. Именно из-за этого бага
> для точной проверки типов (например, в функции `deepEqual`) часто используют надёжный метод
> `Object.prototype.toString.call(value)`.

## 🔵 Как это работает в TypeScript?

В TypeScript `undefined` и `null` — это не просто значения, а **полноценные самостоятельные типы** (`type undefined` и
`type null`).

### 1. Строгий режим (`strictNullChecks`)

В современном TypeScript всегда включена настройка `strictNullChecks: true`. Это значит, что TypeScript берёт на себя
защиту от ошибок "пустоты". Ты больше не можешь случайно положить `null` в переменную, которая ожидает только строку:

```typescript
// ❌ Ошибка компиляции: Type 'null' is not assignable to type 'string'.
let userName: string = null

// ✅ Правильно: нужно явно разрешить пустоту через Union Type
let safeUserName: string | null = null
```

### 2. Опциональные параметры и свойства (`?`)

Когда ты ставишь знак вопроса в интерфейсах или параметрах функции, TypeScript под капотом автоматически добавляет тип
`| undefined`.

```typescript
interface User {
  name: string
  age?: number // Под капотом TypeScript понимает это как: number | undefined
}

const u: User = { name: 'Anna' }
// TypeScript знает, что u.age может быть undefined, и заставит тебя это проверить
```

**Важное правило:** Знак `?` добавляет _только_ `undefined`. Если ты ожидаешь, что с бэкенда может прийти `null` (в
базах данных пустота обычно хранится как `NULL`), тебе нужно прописывать это явно: `age: number | null`.

### 3. Специальные операторы для работы с пустотой

TypeScript (совместно с современным стандартами JS) даёт удобные операторы, чтобы не писать длинные проверки
`if (x !== null && x !== undefined)`:

- **Опциональная цепочка (`?.`)**: Безопасно обращается к вложенным свойствам. Если слева окажется `null` или
  `undefined`, код не выбросит ошибку "Cannot read properties of undefined", а тихо вернёт `undefined`.
  ```typescript
  const city = user?.address?.city
  ```
- **Оператор нулевого слияния (`??`)**: Подставляет значение по умолчанию **только** если слева `null` или `undefined`.
  Это намного безопаснее классического `||`, который срабатывает вообще на любой falsy-мусор (`0`, `""`, `false`).
  ```typescript
  const score = user.score ?? 100 // Если score === 0, то вернётся 0, а не 100!
  ```
- **Non-null assertion (`!`)**: "Клятва" компилятору. Означает _"Эй, TS, заткнись, я на 100% уверен, что здесь точно нет
  null или undefined"_. Использовать нужно редко и с огромной осторожностью, так как это отключает защиту TS на этой
  строчке.

  ```typescript
  const element = document.getElementById('root')! // "Я клянусь, что этот div есть в HTML"
  user.name!.toUpperCase() // "Я клянусь, что name - это строка"
  user.age!.toFixed(2) // "Я клянусь, что age - это число"
  ```

  **Важное правило:** Знак `!` добавляет _только_ `undefined`. Если ты ожидаешь, что с бэкенда может прийти `null` (в
  базах данных пустота обычно хранится как `NULL`), тебе нужно прописывать это явно: `age: number | null`.
