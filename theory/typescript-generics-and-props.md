# TypeScript Generics, Props & Readonly в React

В этом документе разобран синтаксис типизации пропсов в React-компонентах с использованием дженериков (generics), ограничений (constraints) и утилиты `Readonly`. 

Разбор проводится на примере следующего кода:

```tsx
export default function DataTable<T extends { id: number }>({
  data,
  columns,
}: Readonly<{
  data: Array<T>;
  columns: Columns<T>;
}>) {}
```

---

## 1. Generic и Ограничения (Constraints)

Объявление generic-параметра происходит прямо на компоненте:

```tsx
export default function DataTable<T extends { id: number }>({ data, columns }) {}
//                              ↑                         ↑
//                         открываем                 закрываем
//                      generic-скобки            generic-скобки
```

### Разбор посимвольно:
*   **`< >`** — Угловые скобки. Зона объявления generic-параметров функции.
*   **`T`** — Имя типа-переменной (placeholder). Это как переменная, но для типов.
*   **`extends`** — Ограничение (constraint). Означает "является подтипом" или "имеет хотя бы эти поля".
*   **`{ id: number }`** — Минимально необходимая форма объекта.

**Суть `extends { id: number }`:**
`T` может быть **любым типом**, но TypeScript пропустит его, только если внутри этого типа **гарантированно** есть поле `id` типа `number`.

**Зачем это нужно:**
Если внутри компонента мы попытаемся обратиться к `item.id`, TypeScript не выдаст ошибку, так как благодаря ограничению он уверен, что `id` там есть. Без `extends` `T` мог бы быть строкой или числом, у которых нет поля `id`.

### Как тип `T` протекает дальше:
```tsx
function DataTable<T extends { id: number }>({
  data,
  columns,
}: {
  data: Array<T>;      // ← T здесь = тип одного элемента массива
  columns: Columns<T>; // ← T здесь = тот же тип, прокидывается в Columns<>
}) {}
```
TypeScript автоматически выводит тип `T` на основе переданных пропсов (например, из массива `users`). Тот же самый выведенный тип `T` затем применяется и для проверки свойства `columns`. Это гарантирует, что колонки соответствуют структуре передаваемых данных.

## 2. Деструктуризация и Типизация аргумента функции

Весь блок `({ data, columns }: Readonly<{ ... }>)` — это **один аргумент функции**. Запись делится на две части: то, что мы деструктурируем, и то, какой у этого тип.

### Левая часть (Деструктуризация): `{ data, columns }`
Вместо того чтобы принимать объект `props` и писать `props.data` внутри компонента, мы сразу "достаем" нужные поля прямо в параметрах функции.

### Правая часть (Типизация): `: Readonly<{ data: Array<T>; columns: Columns<T> }>`
Это описание структуры объекта, который мы только что деструктурировали:
*   **`data: Array<T>`**: поле `data` должно быть массивом элементов типа `T`.
*   **`columns: Columns<T>`**: поле `columns` должно быть типом `Columns`, который внутри себя использует тот же тип `T` (например, чтобы понимать, какие ключи можно выводить в таблице).

---

## 3. Зачем нужен `Readonly<...>` и почему нельзя менять пропсы

`Readonly` — это встроенный утилита-тип в TypeScript, который делает все поля объекта доступными только для чтения. Их нельзя переназначить внутри компонента.

### Как выглядело бы переназначение без `Readonly`:
Если бы `Readonly` не было, TypeScript позволил бы менять пропсы напрямую:

```tsx
function DataTable<T>({ data, columns }: { data: Array<T>; columns: Columns<T> }) {
  // ПЕРЕНАЗНАЧЕНИЕ ПРОПСОВ:
  
  data = []; // Полностью перезаписываем переменную data
  data.push({ id: 999 } as T); // Мутируем массив
  columns = columns.slice(0, 2); // Перезаписываем колонки

  return <table>...</table>;
}
```

### Почему люди пытаются это сделать?
Часто из-за попытки быстро отфильтровать данные "на лету" без создания новых переменных (например, `data = data.filter(...)`) или попытки задать значения по умолчанию (вместо правильного `function Component({ data = [] })`).

### Почему мутировать пропсы ПЛОХО (Правила React):
В React есть строгое архитектурное правило: **Пропсы должны быть read-only**.

1.  **Родитель не узнает об изменениях:** Если изменить `data` внутри дочернего компонента, React не запустит перерисовку (re-render). UI потеряет синхронизацию с реальными данными.
2.  **Непредсказуемые баги (Side effects):** Если мутировать массив или объект (например, через `push`), который передан по ссылке от родителя, оригинальные данные родителя изменятся. Это сломает логику в других частях приложения, использующих эти же данные.
3.  **Чистые функции:** React требует, чтобы компоненты работали как чистые функции по отношению к своим пропсам (при одинаковых пропсах возвращается предсказуемый UI без побочных эффектов).

### Итог по `Readonly`:
Обертка `Readonly<...>` не влияет на итоговый скомпилированный JavaScript-код. Она работает исключительно как строгий "надзиратель" на этапе написания кода в TypeScript, выдавая ошибку компиляции при любой попытке модифицировать пропсы, тем самым защищая разработчика от грубых архитектурных ошибок.

## 4. Разбор `ColumnDef<T>` и `Columns<T>`

Часто в таблицах конфигурация колонок выносится в отдельные типы:

```tsx
type ColumnDef<T> = Readonly<{
  label: string
  key: string
  renderCell: (row: T) => React.ReactNode
  comparator: (a: T, b: T, sortDirection: SortDirection) => number
}>

export type Columns<T> = ReadonlyArray<ColumnDef<T>>
```

### Что значит `Def`?
`Def` — это сокращение от **Definition** (Определение). `ColumnDef` переводится как "Определение Колонки" (конфигурация одной колонки). Это стандартное соглашение по именованию в React-библиотеках (например, в TanStack Table).

### Разбор `ColumnDef<T>`:
*   **`type ColumnDef<T>`**: Создаем тип для конфигурации колонки, принимающий generic `T` (тип данных одной строки, например `User`).
*   **`Readonly<{ ... }>`**: Конфигурация колонки не должна мутироваться на лету.
*   **`label: string` и `key: string`**: Заголовок колонки и её уникальный идентификатор.
*   **`renderCell: (row: T) => React.ReactNode`**:
    *   Функция, описывающая как отрисовать ячейку.
    *   Принимает строку `row` типа `T`.
    *   Возвращает `React.ReactNode` (любой валидный React-элемент: строку, JSX, `null`).
*   **`comparator: (a: T, b: T, sortDirection: SortDirection) => number`**:
    *   Функция для кастомной сортировки колонки.
    *   Принимает два элемента `a` и `b` (типа `T`) и направление сортировки.
    *   Возвращает `number` (для использования в классическом методе массивов `.sort()`).

### Разбор `Columns<T>`:
*   **`export type Columns<T> = ReadonlyArray<ColumnDef<T>>`**: Это просто массив, состоящий из наших "определений колонок" (`ColumnDef`).
*   **`ReadonlyArray`** гарантирует, что сам массив колонок нельзя будет изменять (через `.push()` и т.д.).
*   Тип `T` прокидывается внутрь каждого `ColumnDef`, связывая типы данных таблицы с методами `renderCell` и `comparator`.

## 5. Дженерики в стрелочных функциях (TSX)

Обычную функцию с дженериком написать просто:
```tsx
function paginateUsers<T>(data: Array<T>, currentPage: number) { ... }
```

Однако при попытке переписать её в виде стрелочной функции в `.tsx` файле возникает проблема:

```tsx
// ❌ Ошибка парсера в .tsx
const paginateUsers = <T>(data: Array<T>) => { ... }
```

TypeScript выдаст ошибку, потому что парсер видит `<T>` и путает его с началом **JSX-тега** (например, как `<div>`), а не с объявлением дженерика.

### Как обойти эту проблему

Существует два стандартных способа указать парсеру, что это именно дженерик:

**Способ 1: Добавить запятую `<T,>` (самый частый паттерн)**
```tsx
const paginateUsers = <T,>(data: Array<T>, currentPage: number, pageSize: number) => {
  // ...
}
```
Запятая однозначно говорит компилятору: "Это список дженерик-аргументов, а не JSX-тег", потому что внутри названий JSX-тегов запятые запрещены синтаксисом.

**Способ 2: Использовать ограничение (extends)**
```tsx
const paginateUsers = <T extends unknown>(data: Array<T>, currentPage: number, pageSize: number) => {
  // ...
}
```
Ключевое слово `extends` (даже если это `unknown` или `any`) также ломает структуру типичного JSX-тега, благодаря чему компилятор понимает истинное назначение конструкции.

> *Примечание: В обычных `.ts` файлах (где JSX не поддерживается) этой проблемы нет, и запись `<T>` работает корректно.*
