# React: Паттерны и лучшие практики

При разработке интерфейсов, таких как таблицы с данными, часто возникают вопросы о том, как правильно управлять состоянием.

---

## 1. Вычисляемое состояние (Derived State)

Одна из самых частых ошибок — дублирование данных в `useState`. 

### Плохо (Антипаттерн):
```tsx
const [users, setUsers] = useState(initialUsers);
const [sortedUsers, setSortedUsers] = useState(initialUsers); // ❌ Дублирование!

useEffect(() => {
  setSortedUsers([...users].sort(...));
}, [users, sortField]);
```

### Хорошо (Best Practice):
Если данные можно вычислить на основе других состояний или пропсов — **делайте это прямо в теле компонента**.

```tsx
const [users, setUsers] = useState(initialUsers);
const [sortField, setSortField] = useState('name');

// ✅ Вычисляем прямо во время рендера
const sortedUsers = useMemo(() => {
  return [...users].sort(...);
}, [users, sortField]);
```

**Почему это круто?**
*   **Меньше багов**: Вам не нужно вручную синхронизировать `sortedUsers` при каждом изменении `users`.
*   **Проще код**: Нет лишних `useEffect` и лишних рендеров (вызов `setSortedUsers` внутри `useEffect` вызвал бы повторный рендер).

## 2. Сброс состояния при действиях пользователя

Когда одно состояние зависит от логики другого (например, пагинация зависит от фильтров или сортировки), важно не забывать сбрасывать зависимые значения.

### Пример с таблицей:
Если пользователь находится на 10-й странице и нажимает на сортировку, список данных полностью меняется. Чтобы он не остался на "пустой" или некорректной 10-й странице, нужно **всегда сбрасывать пагинацию**.

```tsx
const handleSort = (key) => {
  setSortField(key);
  setCurrentPage(1); // 💡 Обязательный сброс на первую страницу
};
```

---

## 3. Логика в обработчиках vs useEffect

Старайтесь выполнять изменения состояния в **обработчиках событий (Event Handlers)**, а не в `useEffect`.

*   **Обработчик события** — это прямое следствие действия пользователя (клик, ввод). Здесь мы точно знаем, *почему* происходит изменение.
*   **useEffect** — это попытка синхронизировать состояние с внешним миром или другими данными. Если вы используете `useEffect` для синхронизации двух внутренних состояний, скорее всего, вам стоит либо объединить их, либо использовать обработчик.

## 4. Математика пагинации (startIndex)

При реализации пагинации на фронтенде часто возникает путаница с индексами.

**Формула:**
```javascript
const startIndex = (currentPage - 1) * pageSize;
const endIndex = startIndex + pageSize;
```

**Почему `(currentPage - 1)`?**
1.  Массивы в JavaScript начинаются с **0**.
2.  Пользователи привыкли видеть страницы, начинающиеся с **1**.
3.  Если мы на 1-й странице, нам нужен индекс `0`.
    *   `(1 - 1) * 5 = 0` (Верно)
4.  Если мы на 2-й странице, нам нужен индекс `5` (при размере страницы 5).
    *   `(2 - 1) * 5 = 5` (Верно)

## 📋 Чек-лист для работы с данными в React:

| Правило | Зачем это нужно? |
| :--- | :--- |
| **Не дублируй в state** | Чтобы избежать рассинхронизации данных. |
| **Сбрасывай страницу** | Чтобы пользователь не видел пустые результаты при фильтрации/сортировке. |
| **Используй useMemo** | Для тяжелых вычислений (сортировка больших массивов), чтобы не тормозить рендер. |
| **Single Source of Truth** | У данных должен быть один источник (например, массив из API), всё остальное — производные от него. |
