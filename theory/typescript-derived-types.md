# TypeScript: Производные типы (Derived Types) - Best Practice?

Создание производных типов на основе уже существующих — **это действительно best practice** в TypeScript. Этот подход лежит в основе написания масштабируемого и надежного кода. 

Однако на вопрос «нужно ли **всегда** так делать?» ответ будет: **нет, не всегда**. Как и в любом архитектурном решении, здесь есть баланс.

Давайте разберем, почему это круто, и когда от этого стоит отказаться.

---

## Почему создавать производные типы — это Best Practice?

Главная цель выведения типов из существующих — **Single Source of Truth (Единый источник истины)** и соблюдение принципа **DRY (Don't Repeat Yourself)**.

Если у вас есть базовая модель, например, `User`, и вам нужна форма для обновления юзера (где все поля необязательные) или список колонок для таблицы юзеров, лучше вывести их из `User`. 

### Плюсы:
1. **Синхронизация:** Если вы добавите поле `age` в `User`, все производные типы обновятся автоматически. Вы не забудете добавить это поле в другие места.
2. **Меньше ошибок:** Снижается вероятность опечаток или рассинхрона типов между клиентом, сервером и UI-компонентами.

### Основные инструменты для этого:
*   **Utility Types:** `Pick`, `Omit`, `Partial`, `Required`.
    ```typescript
    interface User { id: number; name: string; email: string; createdAt: Date; }
    
    // Вместо создания нового интерфейса:
    type UpdateUserDTO = Partial<Omit<User, 'id' | 'createdAt'>>;
    ```
*   **Indexed Access Types & `typeof`**.
*   **Mapped Types:** Создание новых объектов на основе ключей старых.

---

## Когда НЕ СТОИТ делать типы производными? (Исключения из правил)

Иногда излишнее увлечение деривацией типов (так называемая "type gymnastics" или "акробатика с типами") приносит больше вреда, чем пользы. Вот когда лучше создать **отдельный, новый тип с нуля**:

### 1. Когда сущности концептуально разные (случайное совпадение)
Если у вас есть тип `Car { id: number, name: string }` и `User { id: number, name: string }`, не нужно делать `type Car = User`. Даже если сейчас их структура совпадает, в бизнес-логике это разные вещи. Они будут развиваться по-разному. Выводите типы только тогда, когда между ними есть логическая связь.

### 2. Публичные API (Public Interfaces)
Если вы пишете библиотеку или публичный API, который будут использовать другие команды, лучше делать интерфейсы максимально явными и простыми.
```typescript
// Плохо для публичного API (пользователю библиотеки будет сложно понять, что это)
export type Config = Omit<Partial<BaseConfig>, 'internal_id'> & { mode: Mode };

// Хорошо для публичного API (явно, легко читается в автокомплите IDE)
export interface Config {
  mode: Mode;
  name?: string;
  timeout?: number;
}
```
Явные интерфейсы генерируют понятную документацию и легко читаются при наведении мышки в IDE.

### 3. Нечитаемые ошибки TypeScript
Чем больше вложенности в ваших производных типах (особенно при использовании `Infer`, `Conditional Types` и глубоких `Omit`/`Pick`), тем страшнее становятся ошибки. 
Если при ошибке TS выдает вам полотно текста на 50 строк, состоящее из `<T extends...>`, возможно, стоит упростить типизацию и явно прописать интерфейс.

### 4. Производительность компилятора TypeScript
TypeScript — это тоже программа, и ей нужно время на вычисление типов. Очень сложные производные типы могут замедлить проверку типов в вашем IDE и увеличить время сборки проекта.

---

## Итог
*   **Правило по умолчанию:** Делайте типы производными. Если вам нужна "часть юзера" или "юзер без ID" — используйте Utility Types. Если вам нужен тип из конфигурации — используйте `typeof` и Indexed Access.
*   **Останавливайтесь, когда:** Тип становится нечитаемым для человека, IDE начинает выдавать непонятные ошибки, или когда сущности просто случайно похожи, но не связаны логически.
