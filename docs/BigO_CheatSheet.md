# Шпаргалка по Big O (Временная сложность)
> Человеческим языком с жизненными примерами от самого быстрого к самому медленному алгоритму.

---

## ⚡️ O(1) — Константная ("Фиксированная")
**Смысл:** Программе абсолютно наплевать, сколько данных ты ей передашь (1 штуку или 1 миллиард). Она **всегда** потратит одно и то же фиксированное время. Размер данных не имеет значения.

* **Жизненный пример:** Узнать, пустой ли шкаф. Неважно, миллион там книг или одна. Тебе нужен ровно один взгляд (одно действие), чтобы сказать да или нет.
* **В коде:** Обращение к элементу массива по индексу (`array[0]`) или чтение/поиск значения в хэш-таблице (чтение свойства объекта `obj.name` или поиск в Set `set.has(item)`).

---

## 🚀 O(log n) — Логарифмическая («Делим пополам»)
Это **самая желанная сложность** при работе с большими данными. Звучит страшно, но работает гениально: с каждым шагом алгоритм отбрасывает **половину** оставшихся данных.

* **Жизненный пример:** Поиск в отсортированном телефонном справочнике. Тебе нужно найти фамилию "Смирнов". Ты открываешь книгу чётко по центру. Видишь "М". Понимаешь, что "С" дальше. Всю первую половину книги можно смело выбросить! Следующим шагом открываешь центр оставшейся половины. Даже если в книге миллиард страниц, ты найдешь нужное слово всего за ~30 шагов.
* **В коде:** Классический пример — **Бинарный поиск** (поиск в предварительно отсортированном массиве).

---

## 📈 O(n) — Линейная («Пропорциональная»)
**Смысл:** Время работы (или память) растёт **прямо пропорционально** размеру входящих данных (`n`). График выглядит как прямая поднимающаяся линия.

* **Жизненный пример:** Посчитать точное количество книг в шкафу. Одной секундой не обойтись. Придётся брать каждую книгу `(n)` и добавлять в счётчик. Если книг 100 — ты сделаешь 100 действий.
* **В коде:** Практически любой цикл `for` или методы `.forEach()`, `.map()`, `.filter()`, которые вынуждены посмотреть на каждый элемент массива хотя бы один раз от начала до конца.

---

## 🏃 O(n log n) — Линейно-логарифмическая («Быстрая сортировка»)
Это компромисс между линейной O(n) и медленной квадратичной O(n²) сложностями. 

* **Жизненный пример:** У тебя есть огромная перетасованная колода карт. Эффективно отсортировать её можно, разделив на маленькие кучки, сортируя их и затем правильно склеивая обратно (Merge Sort) — это быстрее, чем сравнивать каждую карту с каждой.
* **В коде:** Именно такую алгоритмическую сложность «под капотом» имеют встроенные функции сортировки в языках программирования, например, `array.sort()` в JavaScript_ (Quick Sort или Merge Sort).

---

## 🐢 O(n²) — Квадратичная («Каждый с каждым»)
Начиная с этого момента алгоритмы считаются **медленными**. Количество рабочих шагов возводится в квадрат от размера данных (`n * n`). 

* **Жизненный пример:** В комнату зашли 10 человек. По правилам этикета они все должны пожать руку **каждому** присутствующему. Если людей всего 10, будет 100 рукопожатий (уже долго!). Если в комнату зайдет 1000 человек, им придётся пожать руку 1 000 000 раз. Процесс рискует затянуться на сутки.
* **В коде:** Матрешка из циклов — цикл внутри цикла. 
  ```javascript
  for (let i = 0; i < array.length; i++) {
      for (let j = 0; j < array.length; j++) {
          // Выполнится n * n раз
      }
  }
  ```

---

## 🐌 O(2ⁿ) — Экспоненциальная («Катастрофа»)
Это очень плохо. Алгоритмы с такой сложностью повесят компьютер (или браузер) намертво даже при смешном количестве данных.

* **Смысл:** При добавлении всего **одного** нового элемента во входящие данные, объём работы **удваивается** (возрастает в 2 раза).
* **Жизненный пример:** Подбор пароля к архиву методом "brute-force" (с проверкой всех комбинаций). Каждая лишняя буква в пароле усложняет взлом в несколько раз.
* **В коде:** Наивная рекурсия (без кэширования), где функция на каждом своем шаге порождает ещё по два вызова самой себя. Например, простейшее "школьное" решение задачи про Числа Фибоначчи.

---

### 📊 Резюме (шпаргалка):
1. **`O(1)`** — Топ. Выполняется мгновенно. 
2. **`O(log n)`** — Отлично. Летает даже на огромных данных. 
3. **`O(n)`** — Нормально. Идём один раз по порядку (базовые циклы).
4. **`O(n log n)`** — Нормально для встроенных сортировок `sort()`.
5. **`O(n²)`** — Плохо. Бесит зависаниями уже на тысячах элементов (цикл в цикле).
6. **`O(2ⁿ)`** — Караул. Никогда так не делай в рабочем проекте без острой нужды.
